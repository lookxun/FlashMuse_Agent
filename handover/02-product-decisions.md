# Product Decisions

## 明确确认过的产品方向

用户明确要求：

1. 不按“短剧平台”去做
2. 先只做“简化版即梦”
3. 用聊天方式完成创作
4. 支持生图和生视频
5. 系统要能理解上下文
6. 系统要帮用户优化提示词

## 当前 MVP 范围

保留：

1. 聊天输入
2. 模型选择
3. 上下文理解
4. 提示词优化
5. 图片生成
6. 视频生成
7. 历史记录（后续补）

去掉或延后：

1. 登录
2. 支付
3. 套餐
4. 点数
5. 多人协作
6. 社区分享
7. 短剧项目管理
8. 镜头管理

## 页面语言要求

用户最新明确要求：

1. 项目文件名不要用中文
2. 网页里除了模型名称，其它尽量用中文

## 交互要求

用户希望：

1. 提供一键启动文件
2. 尽量不要有黑窗
3. 点开就能直接测试

## 当前模式交互结论

用户最新确认后，模式结构已调整为三种：

1. `Agent 模式`
2. `图片生成`
3. `视频生成`

当前产品原则：

1. `Agent 模式`负责“帮用户想清楚”，适合目标不明确的用户，应偏智能傻瓜式
2. `图片生成`和`视频生成`负责“帮用户高效执行”，适合已有明确提示词、愿意自己调参数的用户
3. 输入框工具栏统一为 `+` 上传入口、模式自定义上弹菜单、模式参数、发送按钮
4. 全模式都要支持上传图片，当前已完成本地上传 / 粘贴图片预览，最多 5 张；图片会进入对话流并传给 Agent / 图片 / 视频接口
5. 每个历史对话必须独立，草稿、上传图片、生成状态不能串到其它对话
6. 不同历史对话可以并发生成，不能因为 A 会话生成中就禁用 B 会话
7. 明确生图 / 生视频意图优先自动执行，不应只把提示词甩给用户，但短句模糊表达不能乱触发生成
8. 意图判断采用硬规则 + AI 分类 + 本地纠错记忆，后续通过反馈日志持续优化
9. 用户刷新浏览器后应保持在当前历史对话，不应自动跳到第一条
10. 用户发送消息后必须立刻有反馈：普通 Agent 先显示“映造正在思考”，图片 / 视频生成要尽快显示等待卡片

### Agent 模式结论

1. 默认进入 `Agent 模式`
2. Agent 面向模糊需求用户，通过互动逐步推进目标
3. Agent 要会理解剧本、生成剧本、生成文字分镜、整理提示词
4. Agent 已支持明确生图 / 生视频时自动转入对应生成流程，但复杂生成前确认策略后续继续完善
5. Agent 不暴露图片模型和视频模型选择，系统自动选快且够用的默认配置
6. 用户没有明确要求高规格时，图片默认按低成本 / 快速策略，视频默认短时长策略
7. Agent 正文不再输出“下一步调整方向”；所有下一步建议改为独立“引导系统”按钮，显示在反馈操作区下方，不属于正文也不参与反馈
8. Agent 回复当前结构为 `content + suggestions`，正文用于回答，`suggestions` 用于引导用户继续问答或进入影片 / 短剧创作流程
9. 用户简单打招呼或问得很短时，Agent 要短回复，一问一答，不要一次输出多段或多个追问

### 引导系统结论

1. 引导系统不是正文，也不是反馈按钮，是 Agent 回复后的独立创作导航按钮
2. 普通问答阶段：前 2-3 个按钮延展当前问题，后 1-2 个按钮引导到短剧 / 影片创作
3. 创作阶段：前 2-3 个按钮修改当前内容，后 1-2 个按钮推进下一步创作
4. 当前创作流程约定为：故事概念 -> 扩展故事 -> 改成文字分镜 -> 生成主角图片 -> 生成场景图片 -> 做成图片分镜 -> 做成视频
5. 文字分镜到图片分镜必须遵守“一镜一张图”，不能五段分镜只给一张图片；建议逐镜生成，先第一镜，再下一镜
6. 角色图生成后要引导生成三视图；场景图生成后要引导生成多角度参考
7. 如果有多版角色 / 场景 / 分镜图，后续一致性生成应通过 `@资产名` 指定版本
8. 引导按钮已升级成 `{ label, action, assetTargetType }`，点击按钮会把目标资产类型带入生成任务；生成类按钮按 `character_image`、`scene_image`、`shot_image`、`shot_video` 标记
9. 引导系统只显示在最后一条 AI 回复下方；用户点击引导或发送新消息后旧引导消失；用户只是在输入框打字时不消失

### 图片生成结论

1. 图片生成面向目标明确、有提示词的用户
2. 用户输入明确提示词时应直接生成图片
3. 提示词不明确时，后续应提示是否先优化提示词，不做长对话引导
4. 图片生成模式要显示图片模型、比例、分辨率等参数；当前已去掉风格按钮
5. 当前图片模型选择只是“普通图片模型”占位，后续补真实模型列表
6. Agent 模式下识别到“生图、出图、做图、画图、生成图片、来一张、人物/角色/男女主/海报/封面/插画等 + 生成意图”会走图片生成
7. `图片生成` 专业模式下用户输入默认就是提示词，不走提示词优化，不显示为右侧用户气泡，而是作为左侧生成结果上方提示词直接显示
8. `图片生成` 专业模式下上传图和 `@资产` 原样作为参考图传给模型；明确 `@` 时只传被引用图并按文本顺序，未 `@` 时传本次上传图
9. 图片结果不显示引导按钮；引导系统只属于 `Agent 模式`
10. 图片同时生成数量 `1张 / 2张 / 3张 / 4张` 已做实，后端会顺序生成并尽量重试补齐
11. 当前可选图片模型列表为 `Seedream 4.5`、`Gemini 3.1 Flash`、`Gemini 3 Pro`、`GPT-5.4 Image 2`；`GPT-5 Image` 因只传比例时仍固定 `1024x1024`、比例不生效，已从页面移除
12. 除 `Agent 模式` 外，用户手动选择的 `图片生成` 模式优先级最高，即使提示词写“视频 / 一段 / 动起来 / 镜头 / 运镜 / 动画”，也必须按图片生成执行，并把需求理解为单帧画面
13. 图片生成结果的“重新生成”只针对当前这一批结果自己的提示词和参数执行，优先使用该 assistant 消息的 `generationMeta.originalPrompt`，没有再用 `message.content`；不能反查别的批次提示词，不能插入用户气泡，不能串到 Agent
14. 图片生成请求已增加 OpenRouter `curl` 兜底；如果 Node `fetch` 返回假 `500 Internal Server Error`，会自动用同请求体 `curl` 重试
15. 图片结果参数显示必须以真实生成文件为准：比例、尺寸、`1K / 2K / 4K` 图标都跟随真实图片像素，不再只显示按钮选择值
16. “智能比例”当前临时规则为当前模型最小可用分辨率的 `16:9`；选择智能比例后，分辨率按钮和尺寸区域淡化，分辨率不可点
17. 图片尺寸矩阵测试结果保存在 `image-size-test-results.md`；当前 OpenRouter 链路下，Seedream 2K 准确，Seedream 4K 退回 2K，Gemini 尺寸固定偏大 1K，GPT-5 基本固定 1024 方图，GPT-5.4 比例生效但 1K/2K 档位不变
18. 2026-05-12 补充规则：用户确认“只传比例、不传尺寸 / 几K”的实测结果作为原生尺寸基准。Seedream 4.5 原生为 `1:1 2048x2048`、`16:9 2560x1440`、`9:16 1440x2560`、`21:9 3024x1296`、`4:3 2304x1728`；Gemini 3.1 Flash / Gemini 3 Pro 原生为 `1024x1024`、`1376x768`、`768x1376`、`1584x672`、`1200x896`；GPT-5.4 Image 2 原生为 `1024x1024`、`1280x720`、`720x1280`、`1568x672`、`1152x864`
19. 2026-05-12 补充规则：Seedream 4.5 分辨率按钮为 `高清2K / 增强4K`，其它图片模型为 `高清1K / 增强2K`。`增强`不是 OpenRouter 原生输出，而是本地 `sharp` 2 倍超分后的最终图
20. 2026-05-12 补充规则：`21:9` 比例放在 `智能比例` 后面，不做超分；选择 `21:9` 时只显示一个原生按钮，Seedream 显示 `高清2K`，其它模型显示 `高清1K`，该按钮要横向占满分辨率区域
21. 2026-05-12 补充规则：图片请求只传 `aspect_ratio`，不再传 `size / 1K / 2K / 4K`；用户在界面看到的比例、分辨率、尺寸和最终生成文件应保持一致
22. 2026-05-12 补充规则：增强档位在按钮上用金色显示；对话流和预览页参数行中，在普通分辨率图标后显示金色 `增强2K / 增强4K`。原生尺寸无需增强标签
23. 2026-05-12 补充规则：超分后只保留增强图，原生临时图删除，避免资产库和对话流里出现两张图
24. 2026-05-12 补充规则：以上图片增强和尺寸规则不要同步到视频生成模式
25. 2026-05-12 最新修正：此前“增强2K / 增强4K”本地超分规则已取消。项目不再做本地超分增强，所有图片均直接使用模型返回原图。
26. 2026-05-12 最新修正：图片请求参数使用 `image_config.aspect_ratio` + `image_config.image_size`。不要再使用旧字段 `size` 判断模型尺寸能力。
27. 2026-05-12 最新修正：图片档位按最新实测开放：Seedream 4.5 为 `2K / 4K`；Gemini 3.1 Flash 为 `1K / 2K / 4K`；Gemini 3 Pro 为 `1K / 2K / 4K`；GPT-5.4 Image 2 为 `1K / 2K`。
28. 2026-05-12 最新修正：`4K` 在按钮和菜单中显示为金色 `超清4K`。对话流和预览页参数行只在当前结果为 `4K` 时显示金色 `超清4K`，`1K / 2K` 不显示额外标签。
29. 2026-05-12 最新修正：如果 OpenRouter 一次返回多张候选图，必须全部显示和入库，不能只取第一张。不同尺寸在同一批里用参数行 `< 1/2 >` 切换尺寸组，默认显示最接近目标尺寸的组。
30. 2026-05-12 最新修正：同一尺寸组内最多显示四张图片，超过四张用图片区域右下角分页按钮切换，不出现横向滚动条。

### 视频生成结论

1. 视频生成面向目标明确、有提示词的用户
2. 用户输入明确提示词时应直接生成视频
3. 提示词不明确时，后续应提示是否先优化提示词，不做长对话引导
4. 视频生成模式要显示视频模型、比例、分辨率、时长等参数；当前已去掉风格按钮
5. 当前视频模型选择只是“普通视频模型”占位，后续补真实模型列表
6. Agent 模式下识别到“生视频、生成视频、图生视频、首帧生视频、动起来、镜头、运镜、动画、来一段视频”等明确表达会走视频生成
7. 单独“来一段 / 写一段 / 搞一段”不再判定为视频，应继续追问或按普通 Agent 处理
8. 视频生成完成后应像图片一样直接显示在对话流里，不只给链接或按钮；视频按比例缩小，鼠标悬停自动播放，移开暂停
9. `视频生成` 专业模式下用户输入默认就是提示词，不走提示词优化，不显示为右侧用户气泡，而是作为左侧生成结果上方提示词直接显示
10. 视频参考图不再默认当作首帧 / 尾帧，不传 `frame_images`；上传图和 `@资产` 只作为普通 `input_references`，是否首帧或尾帧由用户提示词决定
11. 视频结果不显示引导按钮；引导系统只属于 `Agent 模式`
12. 视频模型可选列表为 Seedance、Kling、Veo、Sora；本机实测 7 个模型均可创建并完成任务
13. 视频时长选项必须按模型动态变化：Seedance / Kling 为 `5秒 / 10秒 / 15秒`，Veo 为 `4秒 / 6秒 / 8秒`，Sora 为 `4秒 / 8秒 / 12秒 / 16秒 / 20秒`
14. 视频默认尝试有声音；非 Sora 模型先传 `generate_audio: true`，若音频参数失败自动退回无声音；Sora 不传 `generate_audio`
15. 除 `Agent 模式` 外，用户手动选择的 `视频生成` 模式优先级最高，即使提示词写“图片 / 海报 / 一张图”，也必须按视频生成执行，并把需求补成视频动作与镜头描述
16. 视频结果的“重新生成”只针对当前这一批结果自己的提示词和参数执行，不能串到图片或 Agent；Sora 长时长轮询上限已按 `12 / 16 / 20秒` 放宽
17. 最新确认：视频任务不应因为排队久而由前端自动停掉；只要 OpenRouter / 模型没有返回 `failed / error / expired`，就继续等待。当前前 2 分钟每 10 秒轮询一次，之后每 30 秒轮询一次，不再有固定最大等待次数
18. 最新确认：同一会话可以在上一条任务未完成时继续发送新任务，最多同时挂起 `10` 个任务；达到上限才禁用发送，按钮显示“任务已满”

### 反馈和学习结论

1. AI 回复完成或图片 / 视频生成完成后，底部显示反馈操作区
2. 常规按钮：复制、重新生成、喜欢、不喜欢
3. 文字回复额外显示“回答不对”按钮，图片 / 视频额外显示“要图给视频或要视频给图”按钮
4. “喜欢”和“不喜欢”只能二选一；再次点击可取消
5. “回答不对”和“要图给视频或要视频给图”再次点击可取消
6. 反馈日志保存在本地 `yinzao-feedback-log-v1`，用于后续分析和改进 AI
7. “要图给视频或要视频给图”会沉淀到本地纠错记忆 `yinzao-intent-memory-v1`
8. 后续需要加强：不喜欢原因弹窗、回答不对原因弹窗、模式错了应选目标模式、反馈日志查看页、自动总结优化规则

### 资产管理和 @ 引用结论

1. 资产管理已从占位改为可用页面，进入后左侧历史对话区切换为“我的资产”分类列表
2. 左侧分类为：全部资产、角色图片、场景图片、分镜图片、分镜视频、待分类、回收站
3. “视频资产”分类已取消；明确镜头 / 分镜的视频归入分镜视频，普通无法判断的视频进入待分类
4. 无法明确分类的图片 / 视频一律进入待分类，不再强行塞进角色 / 场景 / 分镜
5. 分类判断优先级为：分镜 > 角色 > 场景 > 待分类；原因是分镜提示词天然会包含角色和场景词
6. 资产卡片小图统一正方形、直角、无圆角；名称在图片内部左下角，底部有从黑到透明的渐变底
7. 右上角文件夹图标用于修改分类，菜单内有对应图标和勾选状态；所有小菜单点击空白处都会关闭；图片和视频的分类菜单必须分开，图片不能改到视频分类，视频也不能改到图片分类
8. 图片资产左下显示 `@资产名`，点击名称等同引用到输入框；视频资产左下只显示名称，不显示 `@` 也没有引用功能
9. 右下角三点菜单用于资产操作，普通资产当前包含重命名、删除；删除不再立刻真删，而是进入回收站；回收站资产菜单里的“删除”改为“恢复”
10. 回收站中的内容保留 30 天，右侧顶部显示红字“回收站中的内容将在30天后删除，不可恢复。”；到期后应自动从本地资产库移除，并同步删除 `public/generated/...` 下对应本地文件
11. 回收站资产右上角不再允许改分类，改为显示剩余删除时间：平时按天、最后一天按小时、最后一小时按分钟
12. `@` 弹窗只显示角色图片、场景图片、分镜图片三个标签按钮，不显示全部、待分类、回收站和视频；标签按钮显示数量，数量为 0 时不可点
13. 输入框提示为“输入文字，上传图片或@资产，描述生成内容...”，其中 `@` 是蓝色可点，点击后等同输入 `@` 并打开引用弹窗
14. 输入框中的 `@资产名` 用蓝色显示；实现方式是透明 textarea 叠加只读 overlay
15. 输入框上方参考图缩略图是用户判断当前参考图的主视觉入口：上传图和 `@资产` 图都显示为 `100x100` 圆角灰边框缩略图，底部黑色渐变内显示 `@名称`
16. 普通上传图只加入缩略图，不自动往文本框插入 `@文件名`；`@资产`、资产管理引用、历史消息缩略图再引用会同时加入缩略图并插入 `@名称`
17. 输入框最多 5 张参考图，上传 / `@资产` / 历史缩略图引用都共用这个上限，超过时显示“最多上传五张图片”
18. 发送后的用户消息下方会横排显示本次参考图缩略图，点击可继续引用；用户文字里的有效 `@名称` 后会显示一个小图，便于确认引用对象
19. 当前资产保存在本地 `localStorage` 的 `yinzao-assets-v1`，后续登录系统上线后应迁移到服务端用户资产库
20. 资产入库命名规则：角色 / 场景有明确名字时用名字，否则用 `角色1`、`角色2`、`场景1`、`场景2`；分镜图片 / 视频优先 `剧名_分镜01_1`，无剧名用 `无名剧01_分镜01_1`；非剧中普通图片 / 视频用 `image01` / `video01`；多版本统一追加 `_2`、`_3`
21. 上传图片入库使用原上传文件名，分类根据文件名 + 本次用户文本判断：分镜 / 镜头进分镜图片，角色 / 人物进角色图片，场景 / 背景进场景图片，无法判断进待分类；同 URL 不重复入库
22. 动物 / 植物主体且无剧中名字时，会从提示词 / 上下文优先提取更贴切名字，如 `小狗`、`彩色荧光猫`、`科幻兔`；当前还不是生成后看图识别

### 输入框富文本后续方案

1. 当前输入框内 `@资产名` 蓝色高亮通过 `textarea + DraftInputOverlay` 双层模拟实现，因为原生 `textarea` 不支持局部文字变色或圆角底
2. 该方案的天然问题是选中长文本、滚动或换行时容易出现“两层字”、选区错位、overlay 与 textarea 渲染不完全一致
3. 用户提供即梦链接 `https://jimeng.jianying.com/ai-tool/home` 并指出即梦输入框内 `@名` 会变色且带图标；调研后判断即梦大概率不是 textarea，而是类似 `Slate.js / contenteditable` 的富文本编辑器，资源里能搜到 `slate` 迹象
4. 即梦方案应作为下一任 AI 接手后的第一优先讨论项：把“是否将输入框重构为 Slate/contenteditable mention 编辑器”方案重新拿给用户确认
5. 如重构，普通文字应是 text node，`@资产名` 是 inline mention node，可局部变蓝、带图标或胶囊底；这样才能真正做到一层文字、选中不重影
6. 重构代价较大：需要重新实现 Enter 发送、Shift+Enter 换行、中文输入法组合态、粘贴图片、撤销历史、光标定位、草稿序列化、引用资产顺序解析和移动端兼容
7. 短期如果不重构，建议接受单层 textarea 稳定方案，输入框内不做局部蓝色，引用区靠上方缩略图 / 胶囊区区分；发送后的消息仍可用 HTML 正常局部高亮 `@资产名`
8. 最新状态：已先完成一版 `contenteditable` 单层输入框重构，不再使用 textarea overlay 显示蓝色 `@资产名`
9. 新输入框必须保留：输入 `@` 按当前光标弹资产菜单、底部 `@` 按钮按当前光标插入、选择资产替换当前 `@xxx`、粘贴图片、Enter 发送、Shift+Enter 换行、发送时按文本中 `@` 顺序解析参考图
10. 输入框文字上限当前为 `2000字`；超过后显示黑色自动消失提示“最多输入2000字”
11. 输入框宽度原则：默认优先 `800px`，高度优先增长；底部工具栏放不下或文字很长时才逐步加宽，最高约 `1006px`；工具栏按钮必须单行显示、文字不能换行、按钮不能重叠

### 参考图生成链路结论

1. 参考图来源统一为输入框上方缩略图、资产库 `@` 图和历史用户消息缩略图
2. 用户明确输入 `@名称` 时，只传这些被 `@` 的图，并严格按文本中 `@` 出现顺序传给生成链路
3. 用户没有明确 `@` 时，才传本次输入框上方全部图片或最近图片
4. 传给模型前必须按 URL 去重，避免同一张图重复占请求体
5. 提示词优化阶段需要看低清预览图来理解人物和场景，否则多参考图容易完全跑偏；当前使用 `512px / 0.72` 预览图，若 413 再退回纯文本
6. 最终生图阶段默认先传原图；只有 OpenRouter 返回 413 时，才用压缩副本重试：先 `1280px / 0.85`，仍失败再 `1024px / 0.78`
7. 公网部署后应优先改为向 OpenRouter 传公网 HTTPS 图片 URL，不再将本地图片转 base64
8. 当前 `bytedance-seed/seedream-4.5` 多参考图一致性仍需观察，如用户继续反馈不参考图，优先加调试信息或切换更强图像编辑模型

### 模型策略结论

1. 顶部右侧对话模型下拉已移除
2. 暂时不做国内外判断
3. 对话模型固定为 OpenRouter 有效 ID：`bytedance-seed/seed-2.0-lite`
4. 默认图片模型：`bytedance-seed/seedream-4.5`
5. 默认视频模型：`google/veo-3.1-lite`
6. 视频当前走 OpenRouter 视频接口，Seedance 独立聚合接口等后续服务器部署时再接
7. 页面测试阶段显示当前对话、图片、视频模型；上线后可以去掉

### 输入框参数面板结论

1. 输入框里的模式切换、画面设置、时长、同时生成数量、`@` 资产弹窗同一时间只允许出现一个，切换另一个或点击输入框空白区域时，上一个必须关闭
2. 输入框弹窗主圆角统一为 `12px`，内部选项卡片约 `8px`，弹窗外层不再使用灰色描边，只保留白底和阴影
3. 画面设置弹窗标题改为“选择比例 / 选择分辨率 / 尺寸”，比例区和分辨率区都使用整块浅灰底板 + 选中白卡浮起的结构
4. 比例顺序固定为：`智能比例`、`16:9`、`4:3`、`1:1`、`3:4`、`9:16`
5. 尺寸展示区会根据当前比例和分辨率联动变化；当前属于前端展示映射，不代表后端真实返回尺寸
6. `图片生成` 模式新增“同时生成数量”按钮，选项为 `1张 / 2张 / 3张 / 4张`；当前已真正接到图片生成接口
7. `Agent / 图片生成 / 视频生成` 三种模式的比例、分辨率、时长、数量状态必须彼此独立；切换模式后，按钮显示应立即刷新为该模式自己的当前值
8. 输入框工具栏所有非蓝色按钮图标和文字统一走浅灰色；只有模式选择按钮保持蓝色
9. 模型按钮当前不依赖 `react-icons` 名称猜测，已直接接入用户指定 CDN 的 `ai-generate-3d-line` 原始 SVG 到本地组件，保证稳定可用
10. 模型菜单和模型按钮图标跟随当前模型厂商变化：OpenAI 用 `openai-fill`，Google 用 `google-fill`，字节用 `tiktok-fill`，可灵等其它使用本地 `ai-generate-3d-line` SVG
11. 输入框模式和参数保存到 `localStorage` 的 `yinzao-input-settings-v1`，刷新后保持当前模式、模型、比例、分辨率、时长和图片数量
12. 模型名较长时按钮自动变长，输入框默认至少 `800px`，需要时随按钮变宽；窄屏时才隐藏按钮文字只留图标
13. 同一会话的生成状态已从单个 `pendingRequest` 升级为 `pendingRequests` 队列；旧 `pendingRequest` 数据仍兼容读取，避免历史本地数据直接丢失

### 媒体结果显示结论

1. 对话内容区最大宽度为 `1006px`
2. 图片生成结果先显示横排缩略图，单张显示区域固定 `250x250px`，四张横排间隔 `2px`
3. 图片缩略图直角显示，内部图片用 `object-contain` 完整居中，不裁切，能看出横图或竖图
4. 图片生成等待卡也为 `250x250px`，四张等待卡并排显示；图片等待卡内部不显示“正在生成图片...”文案和三个点动画，只保留进度和已等待时间
5. 点击图片缩略图会打开全屏媒体预览页；点击视频结果也打开同一预览页
6. 对话框内 AI 消息左侧小 logo 已去掉；左侧栏 Logo 保留
7. 图片生成失败卡必须和成功图片区同风格：灰底、直角、每张 `250x250`；生成几张失败就显示几张失败卡；左上角显示 `emotion-sad-line + 图片生成失败`；真实错误红字显示在卡片下方
8. 视频生成等待 / 成功 / 失败区域必须和图片结果风格一致：单个 `640x360` 灰底区域；等待卡只显示进度和已等待时间；成功视频完整居中显示；失败卡左上显示 `emotion-sad-line + 视频生成失败`；真实错误红字显示在卡片下方
9. 视频失败错误不能隐藏。最近一次失败真实原因是 `InputImageSensitiveContentDetected.PrivacyInformation`，含义是参考图可能包含真人隐私信息，平台拒绝生成
10. 预览页顶部只留空，左右和底部贴边；下层遮罩使用透明黑色，不是白色透底
11. 预览页左侧为独立媒体预览区，右侧为固定提示词区；左侧底层是毛玻璃，媒体本身不加滤镜、不加圆角，图片直角显示
12. 图片预览默认是 `适合尺寸`；`实际尺寸` 必须按图片真实像素显示且百分比为 `100%`；滚轮和 `+ / -` 从当前可见比例继续缩放，范围 `10% - 250%`，缩放后不显示滚动条，用拖拽平移图片
13. 视频预览按适合尺寸完整显示；当前预览是视频时，左上角缩放按钮应显示为透明禁用态，不可点击
14. 预览页右侧文件名下方显示生成参数：模型、比例、尺寸、分辨率图标和视频时长；分辨率如 `1K` 必须使用与媒体参数行一致的图标组件，不要变成普通文本
15. 右侧提示词标题统一为“图片提示词”，前面有信息图标；标题同一行右侧显示黑色小号“使用提示词”按钮，点击后把提示词填入输入框、聚焦并关闭预览页
16. 当前对话流内图片 / 视频媒体总数超过 `2` 个时，左侧预览区右侧显示浮层缩略图列表，不占布局，不影响主媒体可显示最大区域；缩略图为 `50x50`、`5px` 圆角、`2px` 描边，选中描边 `#367cee`
17. 缩略图列表包含当前对话流内所有生成图片和视频；视频缩略图左上角显示视频图标，图片不显示图标；打开预览页后列表必须自动定位到当前选中的缩略图
18. 缩略图列表如果可以完整显示所有项，就不显示上下按钮；只有需要滚动时才显示 `50x50` 上下按钮；列表滚动条必须隐藏，但保留滚轮滚动能力

### 系统提示消息结论

1. 新对话不再自动插入“你好，我是映造...”默认欢迎消息
2. 切换 `Agent 模式 / 图片生成 / 视频生成` 时，在当前对话流插入一条 `role: "system"` 的灰色系统提示消息
3. 系统提示前置图标使用模式切换菜单里的对应图标；“当前已切换到XX模式”加粗，后面同一行接说明文字，不换行
4. 系统提示是普通对话流位置的一条消息，不能固定在最底部；后续用户消息和生成结果必须出现在它下面
5. 只有系统提示、没有用户输入的会话仍算空会话，不能因为只切换模式就不断产生历史里的“新对话”
6. 系统提示不应进入 Agent / 意图判断 / 反馈日志上下文
